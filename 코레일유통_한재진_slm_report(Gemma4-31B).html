01모델 선택 이유

해당 모델은 가장 최근에 Google에서 발표한 가장 주목 받는 최상위권 오픈 웨이트 모델로 내부망에 설치한 AI서버(RTX PRO 6000)를 활용한 내부 개발자들의 VSCode와 연결한 바이브 코딩용 엔진 활용 용도로 선택하였으며 동시에 8명의 개발자가 접속하여 질의를 남겨도 안정적으로 답변 제공이 가능한 31B 모델로 선택하였고, 멀티모달 지원으로 향후 이미지 분석등의 용도 활용이 가능하여 선택하였습니다.

03학습 데이터와 튜닝 방식

사전학습 데이터: Gemma-4는 Google의 웹 크롤링 데이터, 학술 논문(arXiv), 소스코드(GitHub 등), 공개 텍스트 코퍼스를 포함한 다중 소스 학습 데이터로 사전학습되었습니다. 특히 영어 데이터 중심이나 다국어(한국어 포함) 데이터가 균형있게 포함되어 있으며, 학습 토큰 규모는 약 40T(40 trillion) 수준으로 추정됩니다. 공개 벤치마크(Wikipedia, Books, Common Crawl 등)와 Code 데이터셋이 혼합되어 학습되었습니다.

Instruction Tuning & SFT: Base 모델(gemma-4-31b)을 기반으로 Instruction-tuned 버전(gemma-4-31b-it)은 대화형 질의응답, 명령어 수행, 코드 생성 등을 위해 SFT(Supervised Fine-Tuning)를 거쳤습니다. Google의 내부 고품질 명령어 데이터셋과 ShareGPT, FLAN과 같은 공개 Instruction 데이터를 혼합 활용했습니다.

정렬(Alignment) 기법: RLHF(Reinforcement Learning from Human Feedback)와 DPO(Direct Preference Optimization) 기법이 적용되어 모델의 응답이 안전하고 사용자 의도에 부합하도록 정렬되었습니다. 유해 콘텐츠 거부, 공정성, 정확성 측면에서 개선되었습니다.

한국어 지원 및 한계: 다국어 사전학습에 한국어가 포함되어 있으나, 영어 중심 학습의 특성상 한국어 성능은 영어 대비 약 85~90% 수준입니다. 공공기관의 한국어 민원서 작성이나 복잡한 행정 문서 분석이 필요한 경우 LoRA/QLoRA를 통한 한국어 특화 미세조정 고려가 필요합니다.

05모델 파일 구성

config.json: 모델의 아키텍처, 레이어 수, Hidden Size, Attention Heads, Vocab Size, Max Position Embeddings(256,000) 등 모델 구조 전체 정의. 모델 초기화 시 필수 파일.

model.safetensors 또는 model-*.safetensors: 31B 모델의 경우 메모리 용량 때문에 여러 개 파일로 분할됨(예: model-00001-of-00003.safetensors). 각 파일은 모델의 가중치를 바이너리 형태로 저장. SafeTensors 형식은 pickle 보다 안전하고 로딩 속도가 빠름.

tokenizer.json 및 tokenizer_config.json: 입력 텍스트를 토큰으로 변환하는 규칙 정의. Gemma-4는 Tiktoken 기반 토크나이저 사용. 한국어 텍스트의 경우 자모 단위로 분해될 수 있어 토큰 길이 증가 가능.

special_tokens_map.json: <pad>, <eos>, <unk> 등 특수 토큰 정의. 질의응답 모드에서 응답 종료(<eos>) 토큰이 중요.

generation_config.json: 텍스트 생성 시 기본 파라미터(temperature, top_p, max_length, repetition_penalty 등) 정의. VSCode Vibe Coding 사용 시 코드 생성 품질에 영향.

README.md: 모델의 메타정보, 라이선스, 사용 예시, 한계점 명시. Hugging Face 저장소에서 다운로드 시 참고.

06활용 가능 업무

1. 개발자 코딩 지원 (Vibe Coding in VSCode): 현재 구성된 RTX PRO 6000 AI 서버 + Ollama + VSCode 통합 환경에서 8명 개발자의 동시 코딩 질의응답 처리. Python, SQL, JavaScript 등 다양한 언어의 코드 자동완성, 버그 설명, 리팩토링 제안. Gemma-4의 코드 이해도가 우수하여 코레일유통의 Pelican/Django 기반 프로젝트, SQL 쿼리 최적화, 데이터 분석 스크립트 생성에 즉시 활용 가능.

2. 공공민원 자동 분류 및 응답 초안 작성: 코레일유통 민원 시스템으로부터의 비정형 민원문을 입력받아 주제 분류(배송 문제, 상품 품질, 환불/교환, 일반 문의 등)를 자동 수행. 분류 결과에 따라 표준 응답 템플릿과 관련 법규를 RAG(Retrieval-Augmented Generation)로 검색하여 담당자 승인용 답변 초안 자동 생성. 민원 처리 시간 단축 및 응답 품질 균일화 기대.

3. 내부 정책 문서 및 법규 검색·요약: 코레일유통 운영 규정, 인사 규정, 구매 지침 등 방대한 내부 문서를 Gemma-4 + RAG로 학습시켜 직원 질의에 즉시 답변. 

4. 보고서 초안 작성 및 데이터 해석: 월간 판매 현황 데이터, 보안 사고 분석, IT 인프라 현황 리포트 등을 입력하면 Gemma-4가 자동으로 요약, 분석 결론, 의사결정 포인트를 구조화된 보고서로 초안 작성. 특히 복잡한 데이터 테이블을 자연어 설명으로 변환하여 경영진 보고 시간 단축.

5. IT 운영 로그 분석 & 이상 탐지: 네트워크 로그, 서버 접근 로그, 애플리케이션 에러 로그를 Gemma-4에 입력하여 이상 패턴 자동 감지. "이 로그 패턴이 일반적인지?" 질의에 즉시 응답. 사이버 보안 위협 조기 감지 지원.

6. FAQ 자동 생성 및 챗봇 기반 구축:  Gemma-4가 답변하는 챗봇 구현. 기존 고객 문의 데이터를 LoRA 미세조정으로 학습하여 도메인 특화 응답 능력 향상.

7. 한국어 문서 작성 보조 (이메일, 안내문, 공지사항): 직원이 핵심 내용만 입력하면 Gemma-4가 경어, 존댓글, 공식 톤의 한국어 문서로 자동 변환. 다만 전문성이 높은 법률 문서는 검토 필수.

07한계와 주의사항

1. 환각(Hallucination) 위험: Gemma-4는 정확하지 않은 정보를 마치 사실인 것처럼 생성할 수 있습니다. 특히 코레일유통의 정책 규정, 법규 해석, 구체적인 숫자 기반 답변에서 오류 가능성이 높습니다. 따라서 자동 생성 답변은 반드시 인간 검증 단계를 거쳐야 하며, 민원 답변이나 법규 해석은 담당자의 최종 검토 필수입니다.

2. 한국어 성능 한계: 영어 중심 학습으로 인해 한국어 자연스러움, 복잡한 경어 표현, 업무용 전문 용어(예: 유통 관련 용어) 이해도가 영어 대비 10~15% 낮습니다. 한국어 민원서, 공식 안내문 생성 시 품질 편차 발생 가능. LoRA 미세조정으로 개선 필요.

3. 보안 및 개인정보 위험: 내부망(폐쇄 환경)이더라도 모델 생성 과정에서 민감한 정보(고객명, 전화번호, 카드 정보 등)가 로그에 남을 수 있습니다. 민원이나 고객 데이터 입력 시 마스킹 처리 필수. 개인정보보호법 준수를 위해 별도의 데이터 익명화 파이프라인 구축 권장.

4. 도메인 특화 성능 부족: 일반적인 질의응답은 우수하나, 코레일유통의 구체적인 운영 규칙, 제품 카테고리, 가격책정 정책 등을 충분히 학습하지 못했으므로 오답 가능성 높음. RAG 시스템 + 도메인 내부 문서 학습 필수.

5. 실시간 정보 부족: 학습 데이터가 2024년 상반기 수준이므로 최신 정책, 법규 개정, 시장 정보에 대응 불가. 날씨, 주가, 뉴스 같은 실시간 정보 필요 시 별도 API 연동 필수.

76 계산 정확도 한계: 모델은 수치 계산보다는 자연어 생성에 특화되어 있습니다. 복잡한 재무 계산, 통계 분석이 필요한 경우 Python 스크립트나 SQL 쿼리 생성 후 검증 필수. "이 예산 안에서 총 구매 가능 수량"과 같은 계산은 신뢰성 낮음.

7. 라이선스 및 상업적 활용: Apache 2.0 라이선스이므로 상업적 활용 가능하나, Google이 향후 라이선스 정책 변경할 수 있습니다. 장기적으로는 라이선스 모니터링 필요. 내부망 폐쇄 환경에서의 사용은 문제 없으나, 외부 API나 클라우드 서비스로 제공할 경우 Google과의 협의 필요.

8. 메모리 및 운영 비용: RTX PRO 6000(96GB)에서 FP16 로드 시 거의 포화 상태. 동시 다중 사용자 처리(8명) 시 응답 지연 가능성. 속도 개선을 위해 GPTQ/AWQ 4bit 양자화 적용 시 메모리는 20GB 이하로 축소되나 응답 품질 3~5% 저하 예상.

08종합 의견

1. Vibe Coding  : Vibe Coding 지원은 현재 구성된 RTX PRO 6000 환경에서 현재 사용중이며, 개발 생산성 30~40% 향상 기대. 개발 팀(8명)의 코드 작성 속도, 레거시 코드 이해, SQL 쿼리 최적화 지원에 빠른 효과 기대. 

2.  MVP 용 AI  : VIbe Coding을 이용한 간단한 서비스를 제작하여 AI 챗봇, 사규검색 등 테스트 용도로 활용 예정

Gemma-4-31B는 코레일유통의 현재 IT 인프라(RTX PRO 6000)와 조직 규모(8명 개발팀, 공공기관 특성)에 '최적 적합'한 모델입니다. 특히 내부망 폐쇄 환경에서의 보안 우위, Apache 2.0 라이선스의 법적 안정성, 코딩 능력 우수성이 강점입니다.
